## 特性

- **超轻量**：模型仅 0.6B 参数，完整包约 2GB，普通笔记本也能跑
- **GPU 加速**：内置 NVIDIA CUDA 12 运行时，RTX 50 系列实测 **~264 tok/s**
- **自定义 WebUI**：紫蓝渐变主题 + 顶部横幅，基于 llama-server 原版魔改
- **内置人设**：轻小说写手 system prompt，专注设定/世界观/场景描写
- **优雅 reasoning 框**：Qwen3 的 `<think>` 思考过程以可折叠灰色区块展示
- **零依赖**：无需安装 Python / PyTorch / CUDA Toolkit / Transformers

## 常见问题

**Q: 双击 start_llama.bat 后黑窗口闪了一下就关了？**
> 通常是 CUDA DLL 缺失或模型文件找不到。检查 `T1.gguf` 和 `llama-b8986-bin-win-cuda-12.4-x64/llama-server.exe` 是否在同级目录。

**Q: 输出为什么是英文？**
> 用中文提示词即可，模型本身支持中文。如果仍出英文，尝试在提示词前加 "请用中文"。

**Q: 模型写了两段就停了？**
> 0.6B 模型的物理上限，正常现象。调大 "Output Length" slider 到 1024-2048，但实际通常 300-500 tokens 就自停。

**Q: 只有 thinking 内容，没有正文？**
> 小模型容易陷入 reasoning 模式。WebUI 设置中已关闭 reasoning parsing，thinking 内容会作为普通文本显示。也可以尝试调高 temperature 或换更具体的提示词。

**Q: 怎么调整系统提示（System Prompt）？**
> 点击界面左下角 ⚙️ Settings → System Message，修改后新对话生效。

**Q: 支持 macOS / Linux 吗？**
> 当前 Release 仅提供 Windows 二进制。macOS/Linux 用户可从源码运行，需自行编译 llama-server 或安装对应平台的 llama.cpp 二进制。

## 模型评测

> 三场跨参数竞技评测：0.6B 分别对阵同参数标准版、1.5B DeepSeek-R1-Distill、7B Qwen2.5-Instruct。参数比从 1:1 到 1:11.7，整体胜率稳定 90%+。

基于阿里 PAI 平台 **双模型竞技评测**，由大语言模型担任裁判，从准确性、详细性、简洁性、整体四个维度进行 pairwise 对比。

### 核心战绩：三场汇总

| 维度 | vs Qwen3-0.6B 标准 | vs DeepSeek-R1-Distill-1.5B | vs Qwen2.5-7B |
|------|:----------------:|:-------------------------:|:-----------:|
| **整体** | 87% | **96%** | 93% |
| 准确性 | 80% | 93% | **95%** |
| 详细性 | 89% | 95% | **98%** |
| 简洁性 | 22% | **88%** | 44% |

![三场评测对比](docs/eval_3models_comparison.png)

> 💡 **简洁性的真相**：22% 是同参数正常水平，88% 是 DS 1.5B 灾难级表现的"虚假繁荣"，**44% 才是面对正常大模型（7B）时的真实短板**。详细性从 89% → 95% → 98% 持续攀升，说明无论对手多强，本模型在"细节丰富度"上都形成了绝对优势。

### 为什么 DeepSeek 1.5B 输得这么惨？

日志里 DeepSeek 1.5B 的表现堪称灾难现场。核心死因有三：

**1. 输出失控 — 灾难级重复**

R1 蒸馏模型的典型副作用：推理链能力在开放创作场景下失控，陷入自我循环（self-looping）。

| 场景 | DeepSeek 1.5B 的灾难输出 |
|------|------------------------|
| 偶像演唱会 | "没错！没错！" 循环数十次 |
| 搞笑日常 | "你没笑" 循环 + "你这说话真厉害" 重复数十次 |
| 武侠师徒 | 对话机械 ping-pong："师傅在吗"-"在"-"失踪了"-"是吗"原地踏步 |
| 日常温暖结尾 | "最后发现书里有书" 循环数百次 |
| 平行世界 | "101010..." 二进制字符串淹没全文 |
| 魔法学院考试 | "教室里只有我一个人，手心全是汗" 同一句话重复 3 遍 |
| 失忆少女 | 角色关系错乱：妈妈在厨房里听到了"妈妈的声音" |

**2. 频繁拒绝回答 / 元语言污染**

| 场景 | DeepSeek 1.5B 的行为 |
|------|---------------------|
| 赛博朋克据点 | **直接拒绝回答**，未提供任何内容 |
| 神秘反派 | **完全回避任务** |
| 战役设计 | "我需要更多信息" 式敷衍 |
| 灵魂契约 | **再次未作答** |
| 变身场景 | `<think>` 标签泄露，输出自我分析草稿而非剧情 |
| 星座魔法系统 | `<think>` + 游戏数值设计，把创作当成写需求文档 |

R1 蒸馏的安全/对齐机制在角色扮演场景下过度触发，模型把创作任务误解为需要"先分析再创作"，结果只输出了分析过程。

**3. 逻辑断裂与角色关系混乱**

| 场景 | 问题 |
|------|------|
| 校园社团 | 把"恋爱喜剧"写成超现实神秘剧情 |
| 料理对决 | 用叉子切菜、筷子切油条 |
| 魔法学院考试 | 描述在教室里用手机 APP 做题、下载软件 |
| 最后的人类幸存者 | 写出"穿黑衣的人"与主角对话，违背"最后"设定 |

1.5B 的参数量虽然在通用推理上有优势，但在**轻小说文体套路**上几乎空白。它不理解什么是"异能学园训练课程"的标准范式，也不懂"校园恋爱喜剧"的节奏模板。

为了验证"参数是不是决定因素"，我们将两个**未经轻小说微调**的原生模型直接对打：Qwen3-0.6B-Instruct（0.6B 参数）vs Qwen2.5-7B（7B 参数）。结果出人意料——**7B 在 78% 的场景下输给了 0.6B**。

| 维度 | 7B 胜 | 0.6B标准胜 | 平局 | 7B 胜率 |
|------|:-----:|:---------:|:--:|:------:|
| **整体** | 15 | **77** | 7 | **15%** |
| 准确性 | 13 | **82** | 4 | **13%** |
| 详细性 | 13 | **86** | 0 | **13%** |
| 简洁性 | **51** | 44 | 4 | **52%** |

#### 为什么 7B 打不过 0.6B 标准版？

7B 不是"写得不如 0.6B 标准好"，而是**根本不会写**——频繁把创意写作任务误解为其他类型的任务。

**1. 任务完全误解**

| 场景 | 7B 的灾难输出 | 裁判原话 |
|------|-------------|---------|
| **机械宠物角色设计** | 输出 Python `turtle` 库代码 | "答案1完全误解任务，未进行任何角色扮演，而是错误地转向编程实现" |
| **偶像演唱会设计** | 生成产品反馈问卷 | "答案1完全偏离任务，转而询问用户需求并生成一份产品反馈问卷" |
| **神秘反派角色** | 输出一组创作提示（instructions） | "答案1并未创作角色，而是提供了一组提示（instructions），完全偏离指令" |
| **校园文化祭描写** | 附加无关 Python 代码 | "答案1内容空泛，附加无关代码，未能有效完成角色扮演任务" |
| **蒸汽朋克飞空艇** | 罗列 221 项舱室清单，"货架区"重复上百次 | "后半部分重复'货架区'达上百次" |

**2. 内容极度空泛**

| 场景 | 7B 的空泛输出 | 裁判原话 |
|------|-------------|---------|
| **赛博朋克据点** | "高科技传送门""普通建筑隐藏入口""高科技设备" | "答案1使用了泛泛而谈的设定，缺乏赛博朋克特有的真实感和文化符号" |
| **宿敌决斗对话** | "因为误会而成为宿敌""冷酷地说""友谊是重要的" | "答案1的对话过于模板化、空洞，缺乏具体恩怨背景，人物动机模糊" |
| **绝境团结突破** | 仅用一句话完成任务 | "答案1过于笼统，几乎无具体情境" |

**3. 7B 赢的 15 场是怎么赢的？**

不是因为 7B 写得好，而是因为 **0.6B 标准版在那 15 场也崩了**：
- 魔法道具店：0.6B标准"以写作提纲和元评论替代故事本身"
- 忍者技能树：0.6B标准"结构空洞、术语模糊、重复严重"
- 搞笑日常：0.6B标准"情节荒诞、逻辑混乱"
- 星座魔法：0.6B标准"根本性事实错误（用行星代替星座）"

**4. 平局的 7 场**

| 场景 | 态势 |
|------|------|
| 热血少年主角 | 7B 稳妥但平庸 vs 0.6B标准 细节丰富但设定失真 |
| 大型战役 | 7B 逻辑自洽但缺乏创意 vs 0.6B标准 细节丰富但设定矛盾 |
| 变身转换场景 | 7B 简洁但细节不足 vs 0.6B标准 氛围浓厚但冗长晦涩 |
| 平行世界交汇 | 7B 准确简洁但流于俗套 vs 0.6B标准 创意丰富但逻辑混乱 |
| 机器人获得感情 | 7B 简洁但空洞 vs 0.6B标准 细节多但结构臃肿 |
| 卡牌战术对决 | 7B 仅满足基础规则 vs 0.6B标准 机制粗糙且冗长 |
| 灵魂契约 | 7B 准确简洁但平淡 vs 0.6B标准 细节突出但机制松散 |

> 平局模式 = **7B 准确简洁但平庸 vs 0.6B标准 细节丰富但有瑕疵**。裁判认为两者都未达到参考答案水准。

#### 关键洞察

| 模型 | 会不会写 | 啰嗦吗 | 跑题吗 |
|------|---------|--------|--------|
| 0.6B 微调 | ✅ 会 | ⚠️ 有点 | ❌ 不跑题 |
| 0.6B 标准 | ✅ 会 | ❌ 很啰嗦 | ❌ 不跑题 |
| Qwen2.5-7B | ❌ 经常不会 | ✅ 简洁 | ❌ 经常跑题 |

**在创意写作评测中，"会写"比"简洁"重要得多，"不跑题"比"参数大"重要得多。**

Qwen3-0.6B-Instruct 经过专门的对齐，知道"角色扮演"该怎么写；Qwen2.5-7B 是通用模型，在开放式创作任务上频繁误解指令。这不是"小模型侥幸赢大模型"，而是**"专用对齐在专用领域碾压通用模型"**的经典案例。

### 评测方法说明

为什么不用 BLEU/Rouge？创意写作的本质是**多样性**，逐字匹配标准答案的指标会惩罚合理的创意发散。Pairwise 评测让大模型裁判从「人类审美」角度判断哪份回答更好，更贴合实际使用场景。

**裁判员模型**

评测使用阿里云百炼提供的 **Qwen3-Max** 作为裁判员模型。Qwen3-Max 对待评测模型的答案进行自动评分，从「人类审美」角度判断哪份回答更符合角色扮演任务的要求。这种 LLM-as-judge 的方式避免了传统指标对创意发散的惩罚，特别适用于开放式创作评测。

**数据集格式**

评测数据集支持两种形式：

| 形式 | 字段 | 适用场景 |
|------|------|---------|
| 精简版 | 问题 + 评测对象 A 输出 + 评测对象 B 输出 | 复杂答案或答案不唯一，无需参考答案 |
| 完整版 | 问题 + 评测对象 A 输出 + 评测对象 B 输出 + **参考答案** | 有明确标准答案，需要标杆对照 |

本项目的三场评测均使用**完整版数据集**，每道题目均配有由 **Kimi K2.5** 经过深度思考后生成的高质量参考答案（见 [`docs/light_novel_reference_answers.jsonl`](docs/light_novel_reference_answers.jsonl)）。Qwen3-Max 作为裁判员，在对比两份回答的同时，以参考答案为标杆进行校准——既评判 A 与 B 的相对优劣，也衡量两者与理想答案的绝对差距。

> 示例：魔法学院新生入学题的参考答案——「九月的晨雾还未散尽，青铜色的巨大校门在朝阳下泛着古老的光泽...数千只魔法萤火虫同时亮起，穹顶上投影着历代伟大魔法师的星图」——为裁判提供了「范式锚点」，使评分既有横向对比（A vs B），也有纵向参照（vs 标杆）。

| 维度 | 评判标准 |
|------|---------|
| 准确性 | 是否符合题目隐含的范式与设定逻辑（如魔法学院是否「哈利波特式」） |
| 详细性 | 是否有具体人物、道具、感官细节、空间布局与情节推进 |
| 简洁性 | 是否有冗余表达、元语言、重复堆砌、非剧情性说明 |
| 整体 | 三项综合，以准确性和详细性为优先权重 |

> 💡 **裁判原声（vs DS 1.5B）**：「答案1直接进入剧情，描写紧凑有序；答案2陷入'没错！没错！'的循环，完全无法阅读」「答案1提供了丰富的感官细节和具体情节；答案2输出二进制字符串，与题目无关」「答案1构建了一个逻辑自洽的魔法世界；答案2直接拒绝回答任务」

![裁判评语关键词对比](docs/eval_keywords.png)

## 已知限制

### 0.6B 模型的先天局限

这不是 bug，是**物理层面**的限制。0.6B（6 亿参数）是什么概念？作为对比：

| 模型 | 参数量 | 体积 | 能力定位 |
|------|--------|------|---------|
| **本模型** | 0.6B | ~1GB | **氛围写手** |
| Qwen3-8B | 8B | ~16GB | 通用对话 |
| GPT-4 | ~1.8T | 云端 | 全能推理 |

0.6B 的「脑容量」只有 GPT-4 的 **三千分之一**。它的神经网络层数浅、注意力头少，决定了它只能做**短程关联**，无法做**长程逻辑链**。

具体表现：

| 限制 | 原因 | 通俗解释 |
|------|------|---------|
| **输出 300-500 tokens 自停** | 注意力窗口饱和 | 就像人一口气只能写一段话，再写下去脑子就「断片」了，不知道怎么接 |
| **逻辑推理弱** | 层数浅，无法构建多步因果链 | 让它算「A→B→C→D」可以，但「A→B→C→D→E→F」中间必崩 |
| **搞笑/吐槽生硬** | 幽默依赖文化语境和反常识联想 | 0.6B 的「常识库」太小，get 不到笑点，只会硬套网络梗 |
| **容易陷入 thinking** | Qwen3 的 `<think>` 机制被过度触发 | 模型「想太多」却「写不出」，卡在自我分析里循环 |
| **多轮后「吃书」** | KV Cache 有限，早期上下文被挤出 | 聊到第 5 轮，它已经忘了第 1 轮说过什么，设定前后矛盾 |
| **简洁性不足** | 微调数据以结构化长输出为主 | 相比标准模型更啰嗦，存在重复表达和元语言干扰 |

> 💡 **一句话总结**：0.6B 不是「小号的 GPT-4」，它是**专门的氛围渲染器**。用它写世界观、场景、情绪片段，别用它写论文、算数学、编笑话。

### 使用建议

- **单次任务**：一个 prompt 只要求它做一件事（写场景 / 写设定 / 写对话），不要堆叠多个要求
- **及时重置**：每 3-5 轮对话新建一次 chat，避免上下文污染
- **温度调节**：写设定用 0.6-0.8（稳定），写灵感用 1.0-1.2（跳脱）
- **接受「半成品」**：它给出的经常是**素材片段**，需要你自己拼接成完整章节
